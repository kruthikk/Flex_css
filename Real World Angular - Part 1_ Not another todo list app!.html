Real World Angular - Part 1: Not another todo list app!

This is Part 1 of our Real World Angular series. For other parts, see links at the end of the post or go to https://blog.realworldfullstack.io/.

So, what’s a real world app?

Our real world app will have a back-end, ability for users to register and sign-in, different roles for users to interact with the data, monetization features like display ads and in-app purchases, interaction with device features like camera, location or contact list, report and stats, analytics.

What are we building?

We’ll be building a trivia game. Here are a couple of highly rate ones on the App Store -

Trivia Crack - https://itunes.apple.com/us/app/trivia-crack/id651510680?mt=8

Quiz Up - https://itunes.apple.com/us/app/quizup/id718421443?mt=8

We will borrow some of their basic features, but really focus on building our own version of it. As they have a more generic version, we’ll focus on a single category for our Questions. We’ll build one focused on technology.

Let’s take a look at the high-level features of the app.

Feature Set 1 - Basic Features

At the end of this feature set we would ensure that the features can be used on web & mobile platforms and we can simulate and test it on our devices -
  • Categories with Questions & Answers and Tags on the QnAs.
  • Ability for users to create account, login, modify/reset password, edit their profile etc
  • Users should be able to add and submit new QnAs
  • Admins can approve users QnAs
  • Game Play - Player (User) can select their preferred category and tags and play against another user.
  • Keep track of the scores
  • Keep track of history for the user’s past games
  • Leader-board

Feature set 2 — MVP

The feature set below will mark our MVP (Minimum Viable Product). Once we have implemented these, it would be enough for us to deploy to the app/play stores. At this stage, we’ll start our alpha/beta testing -
  • Some kind of digital currency/credits that enables user to unlock more features or remove limitation on simultaneous number of games
  • User can either earn or make in-app purchase
  • Display of Ad for revenues
  • Flag incorrect QnA, like button, rating, difficulty level, add more tags. Admin can review incorrect flag and modify/delete the Q.
  • Users can search and add other users as friends
  • Social media integration (share, tweet, like, etc)
  • Ability to share the app with others
  • More game play modes: play with friends, challenge mode
  • Enhanced leader-board

Feature Set 3 - Nice to have

These features will be incorporated once the game goes live. We’ll also open up for user feedback and add to this list -
  • Add more roles like moderator so the QnA approval workflow is self managed. Users can submit their QnA, and moderators can approve it.
  • Enhance credit system for users submitting & moderators approving QnAs
  • Game Modes: Offline game mode, QnA mode (single player), Interview prep mode
  • User can use their camera/microphone to submit QnAs with embedded media

Feature set 4 - Advanced Features / Other Possibilities

Based on user feedback and how much traction we get, we may or may not include these -
  • Location based features, such as connecting with local players
  • Chat engine for people to share messages & media
  • Multi-lingual QnAs
  • Ability for users to share their profile/scores on job/work related sites like LinkedIn, Hired or include profile link on their github/social accounts.

Wow, that’s a lot of work to build an app, so let’s get going.

If you have your own idea about an app, feel free to create your own version of a feature list for your app and follow along this series to build your own app with me. Feel free to reach out to me, or share your experience or provide feedback.

Now that we know what we’re building, let’s continue from where we left off in the prior part of the series …

In Part 0, all we did was create our shell application using the cli. Let’s replace references from rwa to trivia in our package.json, angular-cli.json and other places.

As we build the app, I would create a new branch for each part in the series and push it to the github repo. You can checkout the current branch of the git repo @ https://github.com/anihalaney/rwa-trivia/tree/part-1.

At this point, if you haven’t already, you should go thru with Angular’s QuickStart and Guides and have an understanding of the various building blocks of Angular 2 app - https://angular.io/docs/ts/latest/guide/architecture.html. In order to fully absorb the rest of this part, you would need a basic understanding of templates, components, directives, modules & routes. It’s ok if you don’t completely understand it as we will be introducing more advanced concepts of these in later part of the series.

See references below for more links.

Before we get into the game play, we need a database of Questions & Answers. We’ll start with creating model classes for Questions/Answers and Categories to which they would belong. We would also add tags to the QnAs.

We’ll create a folder model under src/app in our application and add category and question models. As a guideline for our folder structure we would follow the style guide - https://angular.io/docs/ts/latest/guide/style-guide.html. We’ll also move the app.component to it’s own app folder under src/app/components.

When moving files around make sure that import statements are modified, wherever these files are being referenced.
Also, note the use of index.ts in each of the high level folders (components, model, services) that export all the classes in these folders so you don’t have to reference individual files in your import.

Also, let’s add services folder to src/app. While we’re at it modify the angular-cli.json, to use scss as the css style extension (set “styleExt”: “scss”) and rename the externsions of the css files to scss. The src folder will look something like this -

On a different command window, start the project with ng serve to ensure that the application is still compiling and serving on localhost:4200

Now, that we have data served on rest apis let’s add services to fetch the data and components to display the data in lists. I’m gonna add 3 services, one for each Category, Tag and Question to the services folder. CategoryService is listed here -

I’ve excluded the import statements for brevity. You can check the github source for complete code. (See source for category.service, question.service & tag.service)
//CategoryService
@Injectable()
export class CategoryService {
private _serviceUrl = 'http://localhost:3000/categories';  // URL to web api
constructor(private http: Http) { 
  }
getCategories(): Observable<Category[]> {
    let url = this._serviceUrl;
    return this.http.get(url)
               .map(res => res.json());
  }
}
Note the addition of rxjs-extensions to the app folder. We would need references to these extensions as we continue building the app.

Next, we would create components to fetch the data from the services and display the data for each of these.

//CategoriesComponent
@Component({
  selector: 'category-list',
  templateUrl: './categories.component.html',
  styleUrls: ['./categories.component.scss']
})
export class CategoriesComponent implements OnInit {
  categories: Category[];
  sub: any;
constructor(private categoryService: CategoryService) {
  }
ngOnInit() {
    this.sub = this.categoryService.getCategories()
                   .subscribe(categories => this.categories = categories);
  }
ngOnDestroy() {
    if (this.sub)
      this.sub.unsubscribe();
  }
}

And the categories template html (also create templates for questions and tags) -

<h3>Categories</h3>
<ul>
  <li *ngFor="let category of categories">
    <span>{{category.categoryName}}</span>
  </li>
</ul>

And finally let’s add all these to our app.module.

@NgModule({
  declarations: [
    AppComponent, CategoriesComponent, TagsComponent, QuestionsComponent
  ],
  imports: [
    BrowserModule,
    FormsModule,
    HttpModule
  ],
  providers: [ 
    CategoryService, TagService, QuestionService
  ],
  bootstrap: [AppComponent]
})
export class AppModule { }

We should now be able to run this and see all our data being displayed on the front-end.

But, all the lists are coming on a single page one after the other. We’ll split them into their own pages and have links from the home page routed to them. Let’s get on with routes.

Add your routes to app.route.ts to src/app folder -

export const routes: Routes = [
  {
    path: '',
    redirectTo: '/categories',
    pathMatch: 'full'
  },
  {
    path: 'categories',
    component: CategoriesComponent
  },
  {
    path: 'tags',
    component: TagsComponent
  },
  {
    path: 'questions',
    component: QuestionsComponent
  }
];

And let’s modify app.component.html for these routes -

<h1>
  {{title}}
</h1>
<div>
  <ul>
    <li>
      <a [routerLink]="['/categories']">Categories</a>
    </li>
    <li>
      <a [routerLink]="['/tags']">Tags</a>
    </li>
    <li>
      <a [routerLink]="['/questions']">Questions</a>
    </li>
  </ul>  
</div>
<router-outlet></router-outlet>

And finally, let’s include the routes in app.module.ts -

...
imports: [
    BrowserModule,
    // Router
    RouterModule.forRoot(routes), 
... 
  ],

Test the app. We should now be able to route to the different components.

In this part, we saw how to wire up a root module with components, templates with services that fetch data from a backend (even though our backend is a mock one at this point) and route between these components. We also looked at the big picture of what we’re building and have started moving towards our goal.

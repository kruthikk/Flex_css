Real World Angular - Part 0: From zero to cli-ng

This is Part 0 of our Real World Angular series. For other parts, see links at the end of the post or see https://blog.realworldfullstack.io/.

Introduction:

With the launch of Angular 2 late last year (September 2016), there have been a series of tutorials, blogs and articles highlighting one or more features of Angular or how Angular can be used with another library such as firebase or material, etc.

Angular
Angular is a development platform for building mobile and desktop web applicationsangular.io

In this series, we’ll focus on building a real world production app using Angular 2 (or simply Angular) and related patterns, tools & technologies. We’ll start right from the beginning and take it all the way to production with an app on the App store, Play store and a mobile friendly web application.

Since, I’ll be building the application as I write each part of the series, we’ll be picking patterns/technologies on the way that best fit in our application as we go along.

The series will focus on building a real world app that can be used as a basis (or template) for building other applications using the same/similar stack of technologies. We’ll not focus on API documentation or tutorials or how to use a particular feature, rather we’ll put references to such blog posts, tutorials, articles as we go along.

Let’s get right into it:

But wait, before, we get started, it’s important to realize how the Web/JavaScript landscape has changed over the last few years. If you come from a jQuery world, you would need to catch up on some newer concepts around building a web application. If you’re not familiar with these, I would suggest you familiarize with these and have at least a basic knowledge of these before proceeding. If you’re already familiar with these you can skip to the next section.

See references at the bottom of the article for links to learn more on these concepts.

  • ES5, ES6, ES2015, ES2016, ES-Next, ES-whatever …
The pace of new JavaScript (now ECMAScript) features/standards has accelerated over the last 3–4 years. All modern browsers now fully support ES5, but are still trying to keep up with the latest specifications of ES6 (ES2015), ES2016 etc.
  • Transpilers (Traceur, Babel, Typescript)
Transpilers are source-to-source compilers that translates between programming languages that operate at approximately the same level of abstraction. Since browsers don’t support latest specifications, transpilers will help us (developers) write code in ES6/ES2015 and beyond, and transpile the output to ES5 that is supported by all browsers.
  • Polyfills/shims
A polyfill is code that implements a feature on web browsers that do not support the feature.

As browsers implement add support for more and more features, it’ll reduce the need to transpile and pollyfill the developed code.

  • Module system (CommonJS, AMD, UMD, ES2015)
Organize JavaScript code in smaller modules. This allows to encapsulate functionality into smaller testable units of code
  • Module loaders (SystemJS, Webpack)
Load these modules in the browser, either statically or dynamically
  • CSS pre-processors (Sass, LESS, Stylus)
CSS pre-processor allows us to use functions, variables, conditions, includes etc to make css more modular and maintainable
  • Build system (gulp, grunt, Webpack)
A build system helps in transpilation, combining modules, copying source maps, minification and creating environment specific distributions for your application.

Let’s get started:

Hope, you’re now familiar with the current state of JavaScript. If not you can learn as we go along. There are lot of resources online that can help you get up to speed.

We’ll be building our application using TypeScript as our language of choice and sass as the css pre-processor.

Angular CLI
The Angular2 CLI makes it easy to create an application that already works, right out of the box. It already follows…cli.angular.io

cli-ng:

Angular CLI (command line interface) allows you to bootstrap a boiler plate app quickly with a set of commands The CLI provides us with a unit test framework, e2e framework, transpiler, css pre-processor, a web server to serve our application with live-reloading (for development), build system to build the application for various environments (development, production, etc.).

Using the cli allows us to focus on building the application functionality, while the cli takes care of the heavy lifting of providing us with an environment to do our development, testing and production-ready builds.

Angular’s cli is based upon the popular ember-cli and has dependencies on angular, typescript, webpack, node-sass among other libraries.

Note: The angular-cli needs to be installed globally as well as locally with the package, and the versions need to match. This is currently a limitation as if you’re working on multiple projects on a single computer (at least on windows), you would need to get the cli on each of the project on the same version as the globally installed one. This can cause huge issues due to the number of dependencies that the cli has and that it may not work with different versions of those dependencies. Hopefully, the cli team resolves this issue soon.

Install the latest versions of the following:

NodeJS - https://nodejs.org

Npm - installed with node

Git - I would recommend you install git, if you don’t have it already as it’ll allow you to download/clone source code from github and you can have your own source checked into a source control repository.

angular-cli

npm install –g angular-cli

VS Code (or Sublime or Webstorm or an editor of your choice) - https://code.visualstudio.com/

Create a new application:

Check the version of angular-cli

ng --version

Let’s create a new Real World Angular app named rwa

ng new rwa

And let’s make sure it works

cd rwa
ng serve

The ng serve command builds and serves the application on port 4200 by default. To test the app go to http://localhost:4200 and make sure it shows up.

Inspect the application:

Let’s take a moment to see what we just created.

package.json

As you can see, the package.json file lists all the angular runtime dependencies along with rxjs, corejs and zonejs.

RxJS - reactive extensions for JavaScript. Please take some time to familiarize with this library and the Observable pattern as we’ll be making extensive use of this throughout our project. (See references for links)

zonejs - used to persist execution context across async tasks. Angular makes heavy use of it for it’s change detection. https://blog.thoughtram.io/angular/2016/02/01/zones-in-angular-2.html

corejs - provides us with a suite of polyfills, so we can safely use ES5, ES6 and later features in older browsers that do not have native support for these features.

The next set of dependencies are used by the cli to build (compile) the angular application and to provide testing infrastructure. We’ll get into more details of these in our later chapters as we start adding more features to our application.

The rest of the dependencies are for typescript and it’s helper libraries that enable us to develop in Typescript, rather than JavaScript. If you haven’t yet, you should familiarize yourself with Typescript as we’ll use this as our primary language for development.

Folder Structure:

If you inspect the folder structure, you would see three folders - e2e, node_modules and src.

e2e - This is where where our end-to-end tests will go. We’ll dive into this in our later chapters

src -This is where all the client side code, templates, css will reside. We’ll be spending most of our time in here, so take a moment to inspect all the files in here. index.html is our main page that’ll host our entire single page app. main.ts is the file that bootstraps our Angular app, while style.css is the global stylesheet file.

node_modules - This is where all the node modules are installed. At first this looks daunting, but most of these are dependencies of angular-cli, provided for ease of developing, building, testing and deploying your app. Most of these libraries will not be deployed to production with your application.

Alternates:

Although, we would follow best practices and guidelines (IMHO) for building our app, there are alternates to building the application. You can certainly choose these alternates. We won’t be covering these alternates in the rest of our series, so you would have to make adjustments accordingly.

To cli or not to cli - The cli gives us several advantages as mentioned above, but it’s not absolutely necessary to use it. You could create an Angular app without the cli. There is great documentation on the angular site on how to go about doing this - https://angular.io/docs/ts/latest/quickstart.html

Using JavaScript instead of Typescript -If you choose not to use Typescript, you could use Angular with regular JavaScript. The Angular docs provide instructions on how to.

CSS instead of SASS - If you prefer not to use a css pre-processor, you could simply use css or another pre-processor like LESS

SystemJS instead of webpack - The cli comes built in with webpack. You could use SystemJS instead of webpack. The Angular docs cover lot of their examples with SystemJS.

Node/Express instead of webpack as web server - Webpack provides it’s own live server to serve up our single page app and static assets needed by the app. You could use Node/Express for serving up your application and webpack just for building the app. We will re-visit this when it’s time to deploy our application to the server. For development, we’ll simply use webpack.
